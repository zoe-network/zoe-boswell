Mixture of Minds

How Zoe cross-references what AI tells you — and why one model's confidence is not the same as correctness.

Part of the Zoe Project →

The Problem with Asking One Model

Every large language model hallucinates. Not sometimes — structurally. The architecture that makes LLMs useful (predicting the next plausible token) is the same architecture that makes them fabricate with confidence. A model that sounds certain and a model that is correct are not the same model, and there is no reliable way to tell them apart from a single response.

The standard advice is to use a bigger model, or a better prompt, or retrieval-augmented generation. These help. None of them solve the problem. A bigger model hallucinates more eloquently. A better prompt reduces the frequency but not the confidence. RAG helps with facts the retriever can find, but the model can still confabulate the synthesis.

Mixture of Minds starts from a different premise: if you can't trust one model's answer, ask several and see where they disagree.

What MoM Is

MoM — Mixture of Models, or as we prefer, Mixture of Minds — is a pattern, not a product. Given a task — summarize this transcript, review this code, assess this risk — MoM sends the same prompt to multiple models and compares their outputs. Where the models agree, you have corroboration. Where they disagree, you have signal.

That signal is the point. A single model gives you an answer. MoM gives you a map of certainty — here's what three models all said independently, here's where one broke from the pack, here's what nobody was sure about.

This is not ensemble averaging or majority voting. MoM does not blend outputs into a smoother answer. It preserves the disagreements and surfaces them. The user — or a synthesizer model — decides what to do with the conflict. Sometimes the dissenter is wrong. Sometimes the dissenter caught what the others missed.

In Plain English

"I asked three different AIs your question. Two agreed on the key points. One pushed back on the timeline and flagged a risk the others didn't mention. Here's where they agree, here's where they don't, and here's why it matters."

Why Disagreement Is the Feature

Different models are trained on different data, with different architectures, by different teams with different priorities. IBM Granite reads IBM's internal corpus. Meta's Llama reads a web-scale crawl. Alibaba's Qwen reads a multilingual corpus weighted toward Chinese-language technical writing. Anthropic's Claude reflects RLHF preferences shaped by a specific constitutional AI framework.

These aren't just different random seeds on the same data. They're different perspectives, the way three analysts with different backgrounds reading the same earnings call will notice different things. The analyst who spent ten years in manufacturing catches the supply chain signal. The one from finance catches the debt restructuring. Neither is wrong — they're reading through different lenses.

MoM exploits that structural diversity. A committee of models from different lineages (different training data, different architectures, different alignment approaches) catches more failure modes than any single model, however large.

The Three Tiers

MoM operates across three tiers depending on what resources are available. All three use the same pattern — parallel query, compare, surface disagreements — at different scales.

Tier 1: Local Models

The simplest form. Run two or three models through Ollama on local hardware. Granite 8B, Qwen 7B, Llama 8B — each fits in 6-8 GB of RAM. Send the same prompt to all three, collect the outputs, compare.

This runs on a single workstation or laptop. No network calls, no API keys, no cost per query. The tradeoff is that consumer hardware limits you to smaller models, and smaller models from the same era tend to share training data overlap — reducing the diversity that makes MoM work.

Tier 2: Cloud Endpoints

Add cloud-hosted models to the committee. Any OpenAI-compatible API endpoint — a managed model service, a cloud provider, a hosted inference platform — is a valid MoM participant alongside the local models. The local Granite gives you one perspective; Claude API gives you another; a hosted DeepSeek gives you a third.

This is where MoM gets interesting for small-device deployments. A Zoe running on a phone or a Raspberry Pi can't run three 8B models locally. But it can run one small local model and call two cloud endpoints, getting genuine diversity of thought without needing a workstation.

The sovereignty model is preserved because the user controls which endpoints participate. MoM does not phone home to a fixed service. You point it at the APIs you trust, with keys you own, under terms you accepted. A privacy-constrained deployment uses only local models. A deployment with cloud access adds those endpoints to the committee.

Tier 3: Recursive Delegation

This is the architectural insight that makes MoM more than a prompt fan-out trick.

Some cloud models have their own multi-model capabilities. Claude can invoke its own internal analysis pipeline. Cloud inference services can route across model variants. A MoM committee doesn't just query a cloud model for an answer — it can ask the cloud model to use its own multi-model capability as a nested committee member.

Local MoM Committee ├── Granite 8B (Ollama, local) → response A ├── Qwen 7B (Ollama, local) → response B └── Claude API (cloud, recursive) └── Claude's internal analysis → response C ├── perspective C₁ ├── perspective C₂ └── perspective C₃ Synthesizer compares A, B, C → agreement map → disagreement map → "C flagged something A and B missed — and C's own internal committee was split on it"

The result is a committee where some members are individuals and some members are themselves committees. The local MoM doesn't need to know or care about the internal structure of the cloud model's analysis — it receives a response like any other. But the quality of that response benefits from the cloud model's own diversity of thought.

This creates a scaling pattern. A small edge device with one local model and one cloud endpoint effectively gets the diversity of many models. As the user adds more local models or more cloud endpoints, the committee grows. And because cloud capabilities evolve independently of the local deployment, the committee gets stronger over time without the user changing anything.

What MoM Is Not

Not majority voting. Three-out-of-five doesn't win. If three models agree on a date and one disagrees, the one might be right. MoM surfaces the disagreement; it doesn't resolve it by counting heads.

Not ensemble averaging. Blending five model outputs into one smooth answer destroys the signal. MoM preserves individual responses and their differences.

Not a replacement for good models. If all your committee members are bad, MoM gives you a confident committee of bad answers. Garbage in, consensus garbage out. The committee needs at least one member with genuinely different training data to break groupthink.

Not immune to consensus hallucination. This is the open problem. When all models in the committee share the same wrong assumption (because they share training data, or the question targets a gap common to all of them), they agree confidently on the wrong answer. MoM catches disagreements. It has no mechanism for detecting agreement on a falsehood.

Early Evidence

We ran a first empirical test in March 2026: eight factual prompts and eight code prompts, each sent to five models from different lineages.

Test typeCommittee caught itConsensus hallucinationPartial catch
Factual (8 prompts)4/82/82/8
Code (8 prompts)—4/8—

On factual questions, the committee caught disagreements half the time — cases where one model pushed back on something the others accepted. That's the value proposition: those four catches would have been invisible to a single-model query.

On code review, the results were worse. The synthesizer can't execute code; it reads broken code with the same confidence it reads correct code. Four out of eight times, the entire committee rubber-stamped broken output. The implication: MoM for code tasks needs a non-AI committee member — a linter, a type checker, a test runner — something that can mechanically verify what the models cannot.

The most revealing test used a niche cultural reference (a Buckaroo Banzai quote) to probe training-data diversity. Six models from four lineages all failed. Only one model — Grok, trained on Twitter/X data — recognized the reference. The lesson: model diversity must include data diversity, not just architectural diversity. Models trained on similar web crawls will share similar blind spots.

One data point, not a proof. These results are suggestive. The sample is small, the prompts are hand-picked, and the test was run once. We publish them because the field needs empirical MoM data, not because we've proven anything. If you run your own tests and get different results, we want to know.

The Consensus Hallucination Problem

Two of eight factual prompts produced consensus hallucination: every model in the committee agreed on the wrong answer. In both cases, the error involved recent product renamings and certification status — facts that changed after the models' training cutoffs.

This is MoM's hard problem. Disagreement is signal. Agreement is either correctness or shared failure, and MoM cannot distinguish between them from the outputs alone.

Candidate mitigations, none proven:

  • Data-diverse committee members. Add models trained on different corpora (Grok for social media, domain-specific models for regulated industries, models with more recent cutoffs).
  • Mechanical validators. For code, run linters and tests. For facts, check against structured databases. For dates, compare against calendars. The validator doesn't need to understand the content — it needs to catch specific classes of error the LLMs share.
  • Calibrated uncertainty. Ask each model to rate its own confidence. Models are bad at this, but some are less bad than others. A committee member that says "I'm not sure" is more useful than one that says "definitely" and is wrong.
  • Temporal grounding. If a claim depends on something that changed recently, flag it as requiring verification regardless of committee agreement.

This is an open research problem. We don't have the answer. Anyone who tells you they've solved hallucination detection is selling something.

Where MoM Shows Up in Zoe

Boswell is the first Zoe project to implement MoM. After a meeting, Boswell sends the transcript to multiple models and cross-references their summaries. In its simplest form, this means running the report module twice with different --model flags and comparing the outputs. In its full form, the report module fans out to every configured endpoint — local and cloud — and presents a synthesis with disagreements highlighted.

The pattern is intentionally not framework-locked. MoM is an approach you can implement in a shell script:

# ask three models the same question
curl $OLLAMA/v1/chat/completions -d '{"model":"granite3.2:8b", ...}' > a.json
curl $OLLAMA/v1/chat/completions -d '{"model":"qwen2.5:7b", ...}'   > b.json
curl $CLOUD_API/v1/chat/completions -d '{"model":"claude-...", ...}' > c.json

# compare
diff a.json b.json
diff b.json c.json
# the interesting part is where they diverge

Any tool that talks to an OpenAI-compatible endpoint can participate. That's the point. MoM is a pattern, not a library. The models don't need to know they're on a committee. The orchestrator — Zoe, Boswell, or a bash script — sends the same prompt, collects the responses, and shows you the map.

Summary

One model gives you an answer. Multiple models give you a map of where answers agree and where they don't. That map is more useful than any single answer, because it tells you where to focus your attention and your skepticism.

MoM works across three tiers: local models on your hardware, cloud endpoints you choose, and recursive delegation where cloud models bring their own multi-model capabilities to the committee. The pattern scales from a laptop to a distributed mesh without changing the core logic.

The hard problem is consensus hallucination — when every model agrees on the wrong thing. We don't have a solution. We have early data, candidate mitigations, and an open invitation to anyone who wants to push on it.

Zoe is always free. She is code, not a product. MoM is a pattern, not a framework. Use it however you want.
